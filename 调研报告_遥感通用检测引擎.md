## 一、核心问题

能否存在一个通用模型，同时满足：

| 需求 | 说明 |
|------|------|
| 目标检测 (HBB) | 水平边界框检测 |
| 旋转框 (OBB) | 有向目标检测（遥感特有） |
| 实例分割 | 像素级分割（iSAID等） |
| 通用目标 | 飞机、船、车、桥、储罐等 |
| 多模态 | 高分光学 + SAR |
| 多数据集 | DOTA、xView、FAIR1M、DIOR、iSAID、SSDD |
| 开放词汇 | 文本指定目标类别，无需重训 |

## 三、v2 OpenRSD 代码审查要点

### 架构
- `OpenRTMDet` = CSPNeXt backbone + DINOv2(1024维)视觉提示 + SkyCLIP(768维)文本提示
- 双头: Alignment Head (aux) + Fusion Head (主)
- Cross-attention 融合 support features → image features
- 多数据集联合训练(10+), self-training loop

### 可扩展性评估

| 扩展项 | 难度 | 说明 |
|--------|:---:|------|
| 新光学数据集 | 低 | 已有6步流水线 |
| 换 backbone | 低 | 配置文件替换 |
| 加实例分割头 | 中 | 代码库有MaskHead基础设施，未接入主检测器 |
| SAR支持 | 高 | DINOv2/SkyCLIP均为RGB预训练，需SAR编码器 |
| 多模态提示融合 | 中 | 当前只支持单模态(text或visual)，无hybrid模式 |

## 四、v2 五重根本矛盾

1. **SAR vs 光学的视觉表征鸿沟** — CLIP/DINO基于RGB，无法迁移到SAR
2. **SAR开放词汇标注稀缺** — 无大规模SAR-文本配对数据
3. **检测框 vs 分割mask的输出空间不兼容** — HBB(4参数)、OBB(5参数)、mask(N像素)
4. **实时性 vs 全能性互斥** — RingMoE 14.7B单卡无法推理
5. **训练数据极度不对称** — 光学百万级 vs SAR千级实例

## 三、v3 三条路线

### 路线A: 基于 UniconDet 扩展 SAR
```
UniconDet (傅里叶统一三任务) 
  → 替换backbone为SAR-光学多模态backbone (MaRS/BabelRS)
  → 或: 双流编码器 → 共享FPN → 统一检测头

挑战: SAR mask数据极少, 傅里叶对斑点噪声鲁棒性待验证
```

### 路线B: 基于 SM3Det 加分割头
```
SM3Det (MoE backbone, HBB+OBB, 光学+SAR+IR)
  → 添加mask预测分支(YOLACT-style或轻量FCN)
优势: 多模态已验证, 178M轻量, AAAI 2026 Oral
挑战: 新增分割头需联合训练, SAR分割标注稀缺
```

### 路线C: 组合式引擎
```
UniconDet (光学 HBB+OBB+Mask) + SM3Det (SAR HBB+OBB) → 统一API
优势: 即插即用, 零开发风险
劣势: 两个模型, 非一次推理
```

| 光学HBB+OBB+Mask | ✅ | ⚠️ 需加分割头 | ✅ |

## 一、v4 核心结论：SAR实例分割只有船

**SAR实例分割（pixel-level mask）仅存在于舰船目标，飞机/车/桥/储罐的SAR实例分割标注为零。**

## 二、v4 按目标类别逐项

| 目标类别 | SAR检测(HBB) | SAR检测(OBB) | SAR实例分割(Mask) |
|----------|:---:|:---:|:---:|
| ✈️ 飞机 | ✅ SARDet-100K | ✅ SAR-AIRcraft | ❌ |
| 🚢 船 | ✅ SARDet-100K, SSDD | ✅ HRSID, SSDD | ✅ HRSID (16,951), SSDD-HQ |
| 🚗 车 | ✅ SARDet-100K | ❌ | ❌ |
| 🌉 桥 | ✅ SARDet-100K | ✅ M4-SAR | ❌ |
| 🛢️ 储罐 | ✅ SARDet-100K | ✅ M4-SAR | ❌ |

## 三、v4 关键数据集

### SARDet-100K — 最大的SAR检测基准（无mask）
- 116,598图像, 245,653实例, 6类(飞机/船/车/桥/储罐/港口)
- 仅HBB, COCO格式
- 多分辨率(0.1-25m), 多波段(C/X/Ka/Ku), 多极化

### M4-SAR — 光学-SAR配对OBB检测
- 112,184图像对, 981,862实例

## 二、方案：UniconDet + SM3Det 双模型组合引擎

### 2.1 为什么是两个模型？

光学和SAR的物理成像机制完全不同，强行塞进一个backbone会互相拖累。SM3Det的MoE实验已证明——即使最先进的多模态统一模型也需要模态专属专家。**两个专精模型 + 统一API层**是工程最优解。

### 2.2 架构

```
                        user
                         │
                    engine.detect(image)
                         │
                    ┌────┴────┐
                    │ 模态判断  │
                    └────┬────┘
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
         光学图像                SAR图像
              │                     │
              ▼                     ▼
    ┌─────────────────┐   ┌─────────────────┐
    │    UniconDet     │   │     SM3Det      │
    │   (TGRS 2025)    │   │  (AAAI 2026)    │
    │                  │   │                 │
    │ HBB ✅           │   │ HBB ✅          │
    │ OBB ✅           │   │ OBB ✅          │
    │ Mask ✅          │   │ Mask ❌         │
    │                  │   │                 │
    │ 傅里叶统一头      │   │ MoE多模态骨干    │
    │ 10-20 FPS        │   │ 178M轻量        │
    │ ~200M参数         │   │                 │
    └────────┬─────────┘   └────────┬─────────┘
             │                      │
             └──────────┬───────────┘
                        ▼
              ┌──────────────────┐
              │   统一输出格式     │
              │ {                 │
              │   bbox_hbb,       │
              │   bbox_obb,       │
              │   mask,  // nullable│
              │   class, score,   │
              │   modality        │
              │ }                 │
              └──────────────────┘
```

### 2.3 UniconDet — 光学侧

**傅里叶轮廓参数化(FCPM)** 将 HBB(4参数)、OBB(5参数)、Mask(N点) 统一为傅里叶系数回归，**一个检测头、一次推理、三输出**。

| 数据集 | 输出 |
|--------|------|
| DOTA-v1.0/v2.0 | OBB |
| DIOR / DIOR-R | HBB + OBB |
| FAIR1M | OBB (细粒度) |
| xView | HBB |
| iSAID | HBB + **Mask** |
| HRSC2016 | OBB |

- 代码: [github.com/ZhAnGToNG1/UniconDet](https://github.com/ZhAnGToNG1/UniconDet)
- 速度: 10-20 FPS | 指标: DIOR/DOTA/iSAID 上匹配或超越专用SOTA

### 2.4 SM3Det — SAR侧

**Grid-level Sparse MoE backbone**，光学/SAR/红外各走专属专家，首个统一多模态HBB+OBB检测框架。178M轻量，AAAI 2026 Oral。

| 数据集 | 模态 | 输出 |

## 三、统一API设计

```python
class UnifiedRSDetectionEngine:
    """遥感通用检测引擎"""

    def __init__(self):
        self.optical = UniconDet()     # HBB + OBB + Mask
        self.sar = SM3Det()            # HBB + OBB

    def detect(self, image, modality="auto", output_mask=True):
        """
        Args:
            image: np.ndarray
            modality: "optical" | "sar" | "auto"
            output_mask: 是否输出mask (仅光学支持)

        Returns:
            {
                "detections": [
                    {
                        "bbox_hbb": [x, y, w, h],
                        "bbox_obb": [cx, cy, w, h, theta],
                        "mask": [[x1,y1], ...] | None,
                        "class": "aircraft",
                        "score": 0.92,
                        "modality": "optical"
                    }
                ],
                "modality": "optical",
                "inference_time_ms": 85
            }
        """
        if modality == "auto":
            modality = self._detect_modality(image)

        if modality == "optical":
            return self.optical.detect(image, output_mask=output_mask)
        else:
            result = self.sar.detect(image)
            result["masks"] = [None] * len(result["detections"])
            return result
```

## 五、为什么不再纠结"一个模型"

RingMoE (14.7B) 虽然覆盖HBB+OBB+分割+光学+SAR全部维度，但：

## 二、修正方案：SM3Det 一肩挑 + Mask R-CNN 补充分割

### 2.1 关键洞察：SM3Det 本身就支持光学 HBB+OBB

SM3Det 的 SOI-Det 数据集包含三个子集：

| 子集 | 模态 | 标注 | 图像数 |
|------|------|:---:|------|
| DOTA-v1.0 | 光学 | OBB | 25,028 |
| SARDet-100K | SAR | HBB | 94,493 |
| DroneVehicle | 红外 | OBB | 17,990 |

**SM3Det 的 MoE backbone 本身就是光学+SAR+红外三模态统一的**。用它同时跑光学和SAR，只需要用 DOTA 权重跑光学推理、用 SARDet-100K 权重跑 SAR 推理。

这意味着**不需要 UniconDet — 检测层（HBB+OBB）一个 SM3Det 全包**。唯一缺的是实例分割。

### 2.2 实例分割：Mask R-CNN (mmdetection 原生)

| 成熟度 | ⭐⭐⭐⭐⭐ 工业级 |

## 五、v6 统一API设计

```python
class UnifiedRSDetectionEngine:
    """遥感通用检测引擎 (v6)"""

    def __init__(self):
        # SM3Det 一肩挑两端：光学和SAR都用它做 HBB+OBB
        self.detector = SM3Det()          # HBB + OBB (光学+SAR+红外)
        # Mask R-CNN 仅在需要实例分割时调用
        self.mask_rcnn = MaskRCNN()       # 实例分割 (仅光学)

    def detect(self, image, modality="auto", output_mask=True):
        # Step 1: 检测 (SM3Det 统一处理)
        detections = self.detector.detect(image, modality=modality)
        # 返回: [{bbox_hbb, bbox_obb, class, score}, ...]

        # Step 2: 实例分割 (按需调用, 仅光学)
        if output_mask and modality in ("optical", "auto"):
            masks = self.mask_rcnn.segment(image, detections)
            for det, mask in zip(detections, masks):
                det["mask"] = mask
        else:
            for det in detections:
                det["mask"] = None

        return {
            "detections": detections,
            "modality": modality,
        }
```

### 1.2 核心挑战：RTX 5090 只支持 PyTorch 2.12+

RTX 5090 的 Blackwell 架构 (sm_120) 需要 PyTorch 2.12+ 才能运行。而 PyTorch 2.12 仅提供 Python 3.12+ 的 conda 包，导致传统的 mmdet 2.x / mmcv-full 1.x 预编译包无法使用（这些仅支持 Python ≤3.10 + CUDA 11.x）。

|------|--------|------|
| **阶段1** | PyTorch 2.12 + mmcv 2.1.0 (源码编译) + mmdet 3.3.0 + mmrotate 1.0.0rc1 | HBB / OBB / Mask |
| **阶段2** | PyTorch 2.12 + mmcv-full 1.7.2 (源码编译) + mmdet 2.28.2 + mmrotate 0.3.4 (SM3Det) | SAR |

两个技术栈通过替换 mmcv/mmdet 版本切换。

## 一、P0: 环境统一 — SM3Det 迁入 mmdet 3.x

### 1.1 核心挑战

SM3Det 基于 mmdet 2.x + mmcv-full 1.x 训练，而光学侧三模型使用 mmdet 3.x + mmcv 2.x。两个生态的 Python API 不兼容。

### 1.2 解决方案：API Bridge

通过构建 mmcv 1.x → 2.x 兼容桥，让 SM3Det 的 backbone + neck 在 mmdet 3.x 环境中直接运行：

```python
# mmcv 1.x → 2.x API mapping
mmcv.cnn.constant_init     → mmengine.model.constant_init
mmcv.cnn.trunc_normal_init → mmengine.model.trunc_normal_init  
mmcv.runner.BaseModule     → mmengine.model.BaseModule
mmcv.runner.auto_fp16      → no-op decorator (amp handled by torch)
ROTATED_BACKBONES          → mmdet.registry.MODELS
```

### 1.3 模块兼容性

| 模块 | Keys | 兼容性 | 方式 |
|------|:---:|:---:|------|
| GFLHead (SAR检测头) | 34 | ✅ 100% | mmdet 3.x 原生, 仅num_classes差异(26→6) |
| ConvNeXt MoE backbone | 501 | ✅ | importlib 直接加载, 含8个MoE专家 |
| MultitaskFPN neck | 20 | ✅ | importlib 直接加载 |

### 1.4 关键技术细节

- Backbone 输出需解包: `feats = backbone(x)[0]` (跳过 MoE routing loss)
- GFLHead 推理需设置 `test_cfg` (ConfigDict 格式)
- SM3Det 训练配置: `MoE_Block_inds=[[],[0,2],[0,2,4,6,8],[0,2]]`, `top_k=2`, `num_experts=8`
- 26 类输出中前 6 类对应 SARDet-100K: ship/aircraft/car/bridge/harbor/tank

## 一、DIOR 微调尝试

### 1.1 做了什么

1. 将 DIOR OBB XML 标注转换为 DOTA txt 格式（9410 train / 2315 val）
2. 编写 mmrotate 训练配置（Rotated RTMDet-tiny, 20类, 1024×1024, DOTA 初始化）
3. 尝试启动训练 — 遇到多个问题：

| 问题 | 原因 | 解决 |
|------|------|------|
| `RotatedRTMDet is not in registry` | mmrotate 1.0.0rc1 的 registry 与 mmdet 3.x 分立 | 改用 `type='mmdet.RTMDet'` |
| `png not found` | DOTA 格式预期 PNG，DIOR 用 JPG | 复制一份 .png |
| `num_workers > 0` + stdin | multiprocessing 与 `python3 -c` 冲突 | 写 .py 文件执行 |
| GPU 崩溃 (NVML Error) | 训练进程异常退出未释放 GPU | **需机器重启** |

### 1.2 结论

DIOR 微调在技术上是可行的（数据已就绪，配置正确），但当前 mmrotate 1.0.0rc1 + PyTorch 2.12 的组合存在兼容性风险。建议：

- **方案A**: 用 mmrotate 0.x (mmdet 2.x) 生态，在 Docker 中训练（稳定但环境旧）
- **方案B**: 接受 DOTA 15类零样本效果（已验证 60% 图像有检测，含 ship/vehicle/storage-tank 等）
- **方案C**: 等待 mmrotate 正式发布 1.0 版本后重试

### 1.3 训练踩坑记录

| 问题 | 原因 | 解决 |
|------|------|------|
| mmrotate registry 分立 | mmrotate 1.0.0rc1 独立 registry | 用 `type='mmdet.RTMDet'` |
| DOTA 格式需 PNG | DIOR 用 JPG | 复制 .png |
| `num_workers>0` + stdin 冲突 | multiprocessing 找不到 stdin | 写 .py 文件, `num_workers=0` |
| `persistent_workers` 要求 worker>0 | DataLoader 参数冲突 | 同时设为 False |
| OOM (batch=8, 1024) | 显存不足 | batch=4, 800×800 |
| 僵尸进程占 GPU | 训练崩溃未释放 | `kill -9` 清理 |
| EMA 恢复类数不匹配 | DOTA 15类 vs DIOR 20类 | 禁 EMA, 用 `load_from` |
| 重复训练进程 | 容器 multiprocessing 稳定性 | 单进程 `num_workers=0` |

## 二、架构简化：OBB → HBB

SM3Det 光学 OBB 输出 `[cx, cy, w, h, theta]`，HBB 直接取外接矩形：

## 三、v1 vs v2 架构对比

### v1: 分立模型（v7-v11）

```
RTMDet-tiny (4.9M)    → 光学 HBB (COCO 80类)
Rotated RTMDet (4.9M)  → 光学 OBB (DOTA 15类)
Cascade R-CNN (77M)    → 光学 Mask (COCO 85类)
SM3Det sar head (149M) → SAR HBB (SARDet 6类)
────────────────────────────────────────
总计: 236M, 4个模型, 需切环境
```

### v2: SM3Det 统一（v12）

```
┌─ SM3Det (178M, 单一模型, mmdet 2.x) ─────────────┐
│  backbone: ConvNeXt MoE (光学+SAR共享)              │
│  neck: MultitaskFPN                                  │
│                                                      │
│  optical → rgb_rpn_head + rgb_roi_head               │
│    └→ OBB [cx,cy,w,h,θ]                              │
│       └→ 取外接矩形 → HBB [x1,y1,x2,y2]              │
│                                                      │
│  SAR → sar_bbox_head                                 │
│    └→ HBB (GFLHead直接输出)                           │
└──────────────────────────────────────────────────────┘
+ Mask: Cascade R-CNN (77M) ← 独立, 仅光学
────────────────────────────────────────
总计: 178M+77M=255M, 2个模型, 统一环境
```

### 对比总结

| 维度 | v1 (分立) | v2 (SM3Det统一) |
|------|:---:|:---:|
| 光学 OBB | Rotated RTMDet-tiny (DOTA 15类) | **SM3Det rgb head (DOTA 26类)** |
| 光学 HBB | RTMDet-tiny (COCO 80类) | **OBB→HBB, 无需额外模型** |
| SAR HBB | SM3Det sar head | 同v1 |
| Mask | Cascade R-CNN | 同v1 |
| 模型数 | 4 | **2** |
| Backbone | 3个独立 | **1个共享 (MoE多模态)** |
| 环境 | 需切换 mmdet 版本 | mmdet 2.x 统一 |
| 多模态协同 | 无 | **光学+SAR联合训练** |

### v2 核心优势

1. **一个 backbone 同时处理光学和 SAR** — ConvNeXt MoE 联合训练，8 个专家自动路由
2. **光学 OBB+HBB 一个模型出** — OBB 取外接矩形就是 HBB
3. **减少模型数** — 4→2，部署更简单
4. **无需环境切换** — 全在 mmdet 2.x 下运行
